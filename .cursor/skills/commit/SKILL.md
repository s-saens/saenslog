# Commit Message Format (커밋 메시지 규칙)

## 개요

에이전트가 변경 파일을 분석해 여러 개의 커밋 단위로 나누고, 각각에 대해 아래 규칙에 맞는 커밋 메시지를 생성·제안한다.  
사용자가 커밋을 수행하도록 안내하거나, 권한이 주어지면 순차적으로 커밋을 실행한다.

## 커밋 규칙 요약

1. 타입(type): contents, refactor, feat, fix, chore 등 (프로젝트 규칙에 따라 확장 가능)
2. 범위(scope): 변경된 곳의 종류(예: blog, project, music, NavBar, musicPlayer 등)
3. 제목(title): 한 줄 요약 — 간결하고 의미 명확하게
4. 상세 description: 변경 이유, 주요 변경사항, 영향 범위, 참고 파일/주의점
5. 변경 단위별로 여러 번 커밋: 한 커밋에는 하나의 논리적 변경 단위만 포함하도록 권장

## 템플릿

type(scope) 제목

상세 설명...

예시 규칙 적용:

- contents(blog) 글 5개 수정, 10개 추가
- feat(musicPlayer) 재생 컨트롤 동작 개선 및 상태 동기화 수정
- refactor(NavBar) Svelte 친화적으로 구조 재배치 및 스타일 분리
- contents(music) 음악 컨텐츠 20개 추가
- fix(blog) 블로그 페이지 라우팅 관련 버그 수정

## 에이전트용 단계별 지침

1. 변경사항 파악
   - 가능하면 staged 변경을 사용하고, 없으면 `git diff --name-only`로 파일 리스트 확인.

1. 그룹화
   - 파일을 도메인/기능별로 그룹화한다 (예: blog 관련 파일, music 관련 파일, UI 컴포넌트 등).
   - 변경유형(추가/수정/삭제/리팩토링/버그픽스)도 고려해 우선순위 정함.

1. 커밋 메시지 초안 생성
   - 각 그룹마다 템플릿에 맞춰 제목과 상세 설명을 생성.
   - 제목은 50자 이내 권장, 한글/영문 혼용 허용.

1. 사용자 승인
   - 생성된 모든 커밋 메시지를 목록으로 보여주고, 수정 또는 병합(합치기) 요청을 받음.

1. 커밋 실행 및 푸시 (자동 실행: 기본 ON)
   - 기본 흐름: 스킬이 호출되면 생성된 커밋 메시지 목록을 내부적으로 생성한 뒤, 사용자 승인 없이 각 그룹별로 `git add` (관련 파일) → `git commit -m "<message>"`를 순차 실행한다. 모든 커밋이 완료되면 `git push`를 실행하여 원격에 푸시한다.
   - 로그/요약: 실행 전후의 요약(생성된 커밋 메시지 목록, 커밋 해시, 푸시 결과)을 호출자에게 출력한다.
   - 안전장치: 자동 실행을 원하지 않으면 스킬 설정 파일을 사용하여 비활성화할 수 있다(아래 참조).

## 설정 파일 (옵션)

스킬 동작을 제어하려면 리포지토리 루트 또는 홈 디렉터리에 다음 파일을 추가할 수 있다:

`.cursor/skills/commit/config.json` 또는 `~/.cursor/skills/commit/config.json`

예시:

```json
{
	"auto_execute": true,
	"push_after_commit": true,
	"branch": null
}
```

- `auto_execute` (bool): true면 스킬 호출 시 자동으로 커밋·푸시를 실행. false면 스킬은 커밋 메시지 및 실행 스크립트만 출력.
- `push_after_commit` (bool): true면 커밋 후 `git push` 실행.
- `branch` (string|null): 특정 브랜치로 강제 푸시하려면 브랜치 이름 지정, null이면 현재 브랜치 사용.

스킬을 호출하는 에이전트는 가능한 경우 이 설정 파일을 확인하고, 설정 없으면 기본값(`auto_execute: true`, `push_after_commit: true`)을 따른다.

1. 분할 권장
   - 한 커밋이 너무 크거나 의도가 혼재되어 있으면 더 작은 단위로 분할 제안.

## 예시 입력→출력 (사용자 예시)

입력: blog에 글 5개가 수정되고 10개가 추가, musicPlayerPill 구현 수정, NavBar 리팩토링, 음악 컨텐츠 20개 추가, 블로그 페이지 버그 픽스

출력(제안 커밋 5개):

- contents(blog) 글 5개 수정, 10개 추가

  블로그 글 수정(5개)과 신규 글 추가(10개). 주요 변경 파일:
  - src/lib/blog/...

  변경 요약: 문법 수정, 메타데이터 보강, 일부 내부 링크 수정.

- feat(musicPlayer) musicPlayerPill 재생 로직 수정

  재생 상태 동기화 및 이벤트 핸들링 수정. 버퍼링 처리 개선 및 메모리 누수 완화.

- refactor(NavBar) NavBar를 Svelte 친화적으로 리팩토링

  컴포넌트 분리 및 reactive 구문 사용으로 렌더링 최적화. 스타일을 모듈화하여 충돌 줄임.

- contents(music) 음악 컨텐츠 20개 추가

  메타데이터와 커버 아트 포함한 음악 항목 20개 추가. DB 마이그레이션 불필요.

- fix(blog) 블로그 페이지 라우팅 버그 수정

  라우팅 조건에서 null 체크 누락으로 인한 에러를 방지하도록 수정.

## 체크리스트 (커밋 생성 전)

- 타입과 범위가 포함되었는가?
- 제목이 간결한가? (권장: <=50자)
- 상세설명에 영향 범위와 주요 파일이 포함되었는가?
- 한 커밋에 하나의 논리적 변경만 포함되는가?

## 확장 및 사용자 설정

- 허용할 타입 목록을 사용자 설정으로 확장 가능 (예: chore, docs, test 등)
- 커밋 실행 권한: 기본은 사용자 승인 필요. 자동 커밋을 허용할지 옵션으로 제공.
