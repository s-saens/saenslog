일어나서 한 10 페이지 정도 남은 책을 다 읽고, 친구 회사 사무실 가서 열심히 일했습니다. 밥을 같이 먹으면서 일을 그만 둘수도 있다는 밑밥을 깔아 뒀습니다.  

저녁으로 고추바사삭을 맛있게 먹고, 여자친구와 밤 산책을 했습니다. 야경이 좋았습니다.  

그리고 저녁에는 OpenRouter, Groq과 같은 LLM API 프로바이더라는 것을 알게 되었습니다. 기존에 에이전트를 굴리기 위해 로컬에선 어렵고 Anthropic이나 OpenAI의 API나 구독제를 사용하면 너무 비쌀 것 같아서 Runpod 같은 클라우드 GPU를 고려하고 있었는데, 이것 역시 만만치 않은 가격이어서 절망하던 중 한 줄기 빛과 같았습니다.  

LLM API 프로바이더에서도 기존 구독을 사용하는 것보다 비싸다는 말이 있지만, 그건 메이저 프로바이더들과 성능을 비슷하게 맞추는 경우인 것 같습니다. 가령 Qwen3-Coder-Next와 같은 초경량 고성능 모델을 사용한다면 성능에 대한 손해를 크게 보지 않고도 비용은 20배 정도 절감할 수 있어 보입니다.  

그리고 LightLLM이라는, 여러 API 프로바이더를 묶어서 하나의 API처럼 활용할 수 있도록 해주는 서비스도 알게 되었습니다. OpenRouter나 Groq과 비슷한 다른 API 프로바이더를 물어봤더니 제미나이 녀석이 LightLLM도 API 프로바이더인 것 마냥 얘기를 하더라구요. 그런데 찾아보니까 프로바이더이긴 한데, 다른 프로바이더로 연결시켜주는 게이트웨이, 프록시 같은 역할을 하는 구조였습니다. 다시 말하자면 OpenRouter나 Groq을 Hermes같은 에이전트에 바로 연결할 수는 있지만, 그 중간에 LightLLM이라는 단 하나의 API를 통해서 여러 API Provider로 라우팅할 수 있게끔 해주는 개념인 것이죠. 와! 제미나이의 실수가 아니었다면 이 중요한 걸 모르고 삽질을 꽤나 했겠습니다. 실수가 운 좋게 좋은 결과를 낳는 경우가 있는데, AI에게서 그런 상황을 보게 될 줄은 몰랐네요.  

그래서 최종적으로 다음과 같은 구조로 오케스트레이션을 하면 되겠다는 것을 겨우 깨달았습니다.

```multica -> hermes agent -> lightLLM -> API Providers(OpenRouter, Groq...)```

즉, API Provider를 사용하고, 이를 LightLLM에서 통합하고, 이를 hermes agent에서 사용하며, 그걸 또 multica에서 하나의 팀원으로 부려먹는 AI 워크플로우입니다. 조만간 이에 대한 글을 열심히 써볼 예정입니다.
